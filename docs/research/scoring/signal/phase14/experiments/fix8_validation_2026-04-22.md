## §1 Change A — Data-Dominant File Filter

### What was built

New module `engine/argot/research/signal/phase14/filters/data_dominant.py` with `is_data_dominant(file_source, threshold=0.65) → bool`.

**Heuristic**: Parse with tree-sitter. Walk module-level statements **and class-body statements inside top-level class definitions**. For each `assignment` whose RHS is a `list`, `tuple`, `dictionary`, or `set` literal, collect the span's row indices as "data rows". Compute `data_rows / total_nonblank_lines`; return `True` if ratio > threshold.

**Threshold choice — 0.65**: Locale provider files have ratios of 0.85–0.99; normal modules with embedded constants sit at 0.02–0.25. The threshold provides comfortable separation on both sides.

### The class-body discovery (iterative fix)

The initial implementation only walked module-level children. Faker locale files use the pattern:

```python
class Provider(BaseProvider):
    cities: Tuple[str, ...] = ("city_a", "city_b", ...)
    streets: Tuple[str, ...] = ["street_a", ...]
```

Data lives in **class body**, not at module level → initial filter returned `False` for all faker locale files → threshold unchanged at 8.20.

Fix: extend `_extract_data_literal_lines` to descend into top-level `class_definition > block` children and apply the same assignment check. After the fix:
- `faker/providers/address/ar_DZ/__init__.py` → `True` (332 non-blank lines, ~96% data rows) ✓
- `fastapi/openapi/models.py` → `False` (338 non-blank lines, mixed methods + constants) ✓
- `faker/providers/bank/__init__.py` → `False` (131 non-blank lines, mostly methods) ✓

## §2 Change B — Narrow "Generated By" to Comment Nodes

### What changed

`engine/argot/research/signal/phase14/filters/autogenerated.py` — `is_auto_generated()` now parses the source with tree-sitter and queries **only `comment` nodes** (i.e., lines starting with `#`). Markers are checked only against comment text in the head window. String literals and docstrings are completely ignored.

Fallback: if tree-sitter parsing raises an exception, falls back to line-prefix check (`line.lstrip().startswith("#")`) to handle truncated/invalid input without crashing.

**Before**: "generated by" in a class docstring → `True` (false positive)
**After**: "generated by" in a class docstring → `False` ✓; "# generated by" in a comment → `True` ✓

The threshold dropped by ~1.05 but did not reach the hoped-for ~4–5 range. The floor at ~7.15 is set by faker's non-locale calibration hunks. Faker is a domain-specific library: even "normal" modules (generators, proxy objects, decorators) contain patterns rare in the generic model-B corpus (multilingual string manipulation, elaborate pluralisation logic, locale-dispatch code), pushing the max calibration score to ~7.15.

In fix8, it is correctly scored (auto_gen=0, not flagged because its BPE score is below the threshold). **Change B confirmed working.**

### Honest assessment of faker

fix8 makes three improvements:
1. Auto-gen false positive eliminated (bank docstring → no longer suppressed)
2. Threshold reduced 8.20 → 7.15 (locale string tables excluded from calibration)
3. Stage 2 catch rate improved 0% → 38% (walrus, match_case, async_adoption now detectable)

But 5/8 stage2 fixtures remain undetectable at any sensible max threshold because faker's calibration floor (~7.15) is intrinsically above those fixture scores. This is not solvable by filter tuning alone: the problem is that faker's "normal" code (locale dispatch, proxy objects, string generators) uses vocabulary patterns that the BPE model considers unusual, creating a high-surprise calibration set even after removing pure data files.

**Root cause of residual misses**: content-driven. Faker uses the same Python idioms (dataclasses, f-strings, genexpr) as the fixtures, but in modules that are surrounded by unusual-vocabulary code, inflating the calibration threshold above those idiom scores.

### Path to full faker coverage

Options (not implemented in fix8):
1. **p99 threshold for faker**: Stage 2 recall stays at 38% (p99 ~5.66 is below the 5 missed fixtures). No gain.
2. **Per-corpus N reduction**: Smaller N increases instability (per Session R).
3. **Second-level data exclusion**: Filter out calibration hunks that are inside classes with >50% data-literal class attributes, even if the file itself is mixed. This would require a hunk-level (not file-level) data-dominance check.
4. **Explicit faker exclusion list**: Rejected — no hardcoded domain knowledge in prod.
5. **Accept faker as out-of-scope for V0**: Faker is structurally different (multilingual fake-data generator); the scoring problem is qualitatively harder. FastAPI and Rich are the primary targets.

**Recommendation**: Ship V0 for FastAPI and Rich. Defer faker to a separate investigation.
