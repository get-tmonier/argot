## §2: Yellow flags discussion

### faker-js: data-dominance (75% of locale files, 70% of total TS file count)

faker-js has 77 locale directories containing 2,271 hand-written data files (name lists, airline codes, etc.)
plus 691 auto-generated locale index files. These 2,962 locale files represent ~92% of the total 3,227 TS
files. The locale data files are not auto-generated (no `// automatically generated` header), but they are
pure data: arrays of string literals and object literals with fixed keys. The scorer **must exclude these
from training/eval** or precision will collapse — every PR touching a locale adds hunks that are essentially
`'new string added'` or `{ name: 'Foo', code: 'XY' }` with no stylistic signal.

**Recommended adapter behavior**: filter out files matching `src/locales/**/*.ts` from hunk selection.
The non-locale core (`src/core/`, `src/faker.ts`, `src/definitions/`, `test/`) has 159 + 63 = 222 files
with normal application code. Hunk density after filtering drops but remains viable:

- Locale-touching PRs will yield 0 eligible hunks after filtering
- Non-locale PRs are less common but richer; conservative estimate: ~40–50% of faker-js PRs touch
  non-locale code → effective cadence ~40–50 PR/year → N=500 requires drawing from ~3 years of history
  (feasible with full `--depth=2000` clone)

**Verdict**: YELLOW not RED — the core is healthy and this is the expected shape for a locale-data-heavy
library. The adapter complexity is known and bounded.

### faker-js: auto-generated locale indexes (21% of TS files)

The 691 `src/locales/*/index.ts` files are stamped `// automatically generated` and regenerated by
`pnpm run generate:locales`. These barrel-export files have no scorer signal. They should be excluded
alongside the data files under the same `src/locales/**` filter.

### Ink: git depth at shallow-clone boundary

Initial 500-depth clone returned exactly 500, suggesting the true count was higher. A 2000-depth clone
confirmed **772 total commits**. This is above the ≥500 gate but is a relatively lean history for a
library started in 2017. The high hunk density (10.3/PR) compensates: 49 PRs suffice for N=500.

### Hono: hunk pool requires ~94 PRs for N=500

At 5.3 hunks/PR average, reaching N=500 means drawing from ~94 PRs. With ≥100 merged PRs in the last
12 months alone, this is comfortably within the 1-year window. No stability concern.

  stability probe (N=100 seed sweep) before committing to N=500. If pool is confirmed ≥400 after
  filtering, escalate to N=500.
